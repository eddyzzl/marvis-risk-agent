# V2 长线区裁决记录（LT-9 / LT-10 / LT-12 / LT-18）

> 2026-07-03，随 v2-master-backlog 收尾裁决。这四项的共同点：待办的本体是"做出并记录一个决定"，而非一段代码。裁决即完成；若未来触发条件出现，按本文档重开。

## LT-9：OS 级沙箱 vs subprocess+护栏 —— 裁决：**subprocess+护栏为 V2 终态**

理由：
1. V2 的威胁模型是单机单用户（关键架构决策既有拍板）：防的是失控资源占用与意外越权，不是恶意租户。namespace/chroot/容器沙箱防的是后者，成本（macOS 无 cgroup、依赖矩阵、调试复杂度）远超收益。
2. 护栏已从"纸面 mock"升级为真进程验证（TST-4，`677064bb`）：真 OOM 进程树击杀+审计 peak、含孙进程的进程组杀、子进程环境白名单真回显、notebook kernel 真杀，且做过"故意弄坏杀逻辑"的检测力自证。INV-6 的执行面证据链完整。
3. GAP-5（loopback 全信任=装插件任意代码执行）已单列治理，与沙箱本体解耦。

触发重开的条件：V3+ 引入多用户/远程访问（见 LT-18 蓄水池），沙箱升级为准入项。

## LT-10：legacy live-session 路径 —— 裁决：**正式定为 triple-opt-in legacy-only 终态**

现状（pipeline.py:156-161）：默认隔离执行；要走 legacy live kernel 必须同时满足 `notebook_isolated_execution=False` + `allow_legacy_live_notebook_execution=True` + 环境变量 `MARVIS_ALLOW_LEGACY_LIVE_NOTEBOOK_EXECUTION=1`，缺一即 typed error。

理由：三重门槛已把误入概率压到几乎为零；worker RPC 化只服务这条本已弃用的路径，是负收益工程。裁决：不做 RPC 化，本路径不再接收功能投入，仅保安全修复；文档语义=「调试后门，生产禁用」。

## LT-12：row-level 聚合去重策略 —— 裁决：**不触发，维持现状**

该项自带触发条件（"仅当业务要求必须保留真实行时"）。当前 JOIN 去重语义（first/last 策略+强制确认门+审计）已覆盖既有全部业务口径，无任何任务要求行级保真去重。裁决：关闭；未来出现该业务要求时以新 spec 重开（锚定 join 引擎设计文档的键字典与指纹机制）。

## LT-18：V3+ 方向蓄水池 —— 裁决：**建池，显式不做项如下**

超出"单机单用户"边界、V2 一律不做，进入 V3+ 评估池：
1. 多用户/权限体系（认证、角色、行级权限）——连带 LT-9 沙箱升级、审计的用户维度。
2. 部署打包（Docker/安装器/自动更新）与远程访问模式（反向代理正式支持已有雏形：MARVIS_TRUSTED_PROXY_HOSTS，扩为产品化是 V3 事项）。
3. 外部决策引擎对接（策略导出到规则引擎格式：Drools/自研 DSL）；PMML 之外的模型服务化（REST scoring service）。
4. 多机资源调度（训练任务分发）、模型注册中心跨实例同步。
5. 实时/准实时监控数据接入（S5 拍板为手动触发，自动化调度是 V3 项）。

进入 V3 评估的门槛：用户显式提出对应场景需求。本池不承诺顺序与时间。
