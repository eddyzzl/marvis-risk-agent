## Part A：全量读码结论

### A1. 总体判断

先说好消息，而且是实证的好消息：**这个项目的工程过程是诚实的**。15 项最高风险的 backlog ✅ 逐项读码核验，14 项完全落地、1 项 partial（残留是两处无行为差异的常量硬编码）——"✅ 不等于钉牢"的担忧被证伪了，commit 说做了的事代码里真的做了（A6）。JOIN 的两条头牌不变量（行数守恒断言+回滚、三层强制确认）是真执行不是装饰，没有找到绕过路径（A4）。建模方法学相比 6-28 审计有大幅实质改善：六算法家族全部两阶段调参、预处理 train-only 拟合门、OOT 按政策隔离（A5）。

坏消息也很具体：**穿过 2000+ 测试、三轮审查、双收官循环活下来的 bug，全部是同一个物种——代码完全按设计运行，但设计对真实数据的语义假设是错的，或者呈现给人的证据与计算的事实不一致**。本轮经对抗验证确认 6 条高危（A2）+ join 专项 1 critical 2 high（A4）：

为什么这类 bug 能全身而退？因为**整个质量体系的每一层（测试、审查、backlog、收官循环）都在验证"代码是否与设计一致"，没有一层在验证"设计是否与数据一致"**。测试喂的是符合代码假设的合成数据，审查对照的是 spec，backlog 核验对照的是 claim——它们共享同一套假设，所以共享同一个盲区。这个判断直接回答了你的第 1、2 问（Part B）。

### A2. 经对抗验证确认（CONFIRMED）的发现
以下 6 条来自子系统读码，每条经 2 个独立怀疑者镜头对抗验证后仍成立；另有 join 专项审计的 critical/high 见 A4。

#### A2.1 [HIGH] Vintage cum_bad_rate only correct for 'newly bad at this MOB' labels; status-flag labels silently inflate then clip at 1.0
- 位置：`marvis/packs/strategy/vintage.py:26`（子系统：strategy-analysis；验证结论：CONFIRMED）
- 细节：vintage_curve consumes marvis/validation/vintage.py compute_vintage_curve, whose cum_bad_rate sums each MOB's bad_count into a running numerator (validation/vintage.py:111-118). That is only correct if bad_col marks the increment (loan newly bad at that MOB, counted once). With the most common panel shape -- a per-snapshot status flag where a bad loan stays bad=1 at every later MOB -- a loan bad from MOB3 onward is added 10 times by MOB12, so cum_bad_rate races toward 1.0 and is silently clipped (warning only fires when the raw ratio exceeds 1.0, so moderately inflated values pass clean). tool_vintage_curve then feeds this into vintage_summary's at_ref/trend ('deteriorating'/'improving'), a decision-facing risk read. Nothing at the tool boundary documents or validates the label semantics (there is no id column to validate against). This matches the maintainer's own memory note that the cum_bad_rate semantics is a known unfixed prerequisite.
- 测试覆盖：tests/validation/test_vintage.py locks the incremental-label semantics with hand-built increment data; no test feeds status-flag (ever-bad) input, so the failure mode is uncovered

#### A2.2 [HIGH] expected_loss_estimate total_el sums point-in-time EL across every historical snapshot month, double-counting the same loans
- 位置：`marvis/packs/analysis/loss.py:198`（子系统：strategy-analysis；验证结论：CONFIRMED）
- 细节：_el_by_month computes EL = balance * P(absorb within horizon) * lgd for EVERY snapshot month in the dataset and total_el accumulates them (loss.py:190-199). For a 12-month panel of the same portfolio with stable balances, total_el is ~12x any economically meaningful portfolio EL, because a loan destined to default is counted in each month it appears. tool_expected_loss_estimate returns this number verbatim and report.py gate_summary_payload promotes it as highlights['total_el'] (the headline number in the portfolio gate and xlsx report). Per-month rows are individually fine; only the cross-month sum is misleading, and no red flag or assumptions entry explains the summation basis.
- 测试覆盖：none found -- tests/test_analysis_pack.py only asserts chain probabilities; tests/test_portfolio_api.py passes total_el through assemblers without locking its computation

#### A2.3 [HIGH] Sentinel-value masking is not recorded in the preprocessing replay chain -> train/serve skew for the sentinel population
- 位置：`marvis/packs/feature/tools.py:616`（子系统：packs-misc-feature；验证结论：CONFIRMED）
- 细节：tool_impute_missing / tool_cap_outliers / tool_normalize / tool_woe_encode all mask sentinel_values (e.g. -999) to NaN before fitting AND before applying to the derived dataset, but the persisted sidecar steps carry only fill_values/bounds/scaler_params/woe_maps with no sentinel info (impute step at tools.py:616; no 'sentinel' step kind exists in marvis/feature/preprocessing.py). At scoring time apply_preprocessing_steps (_apply_impute L141, _apply_cap L169, _apply_normalize L189, _apply_woe L220, _apply_missing_indicator L153 -- consumed by marvis/packs/modeling/scoring.py and the handoff notebook) treats a raw -999 as a genuine value: impute leaves it as -999 instead of the median, cap clips it to the lower bound, normalize scales it to an extreme z-score, numeric WOE assigns it the lowest bin's WOE instead of na_woe, and the __was_missing indicator flips from 1 to 0. Exactly the 'no bureau hit' population the user explicitly confirmed as special gets systematically wrong scores in production replay. grep of tests/: sentinel tool tests (tests/test_feature_pack.py:1328) and replay round-trip tests (tests/test_feature_pack.py:1024,1142; tests/test_modeling_pack.py:3436) exist separately, but no test combines sentinel_values with chain replay -- none would catch this.
- 测试覆盖：none found

#### A2.4 [HIGH] slice_aggregate bad_rate/approval_rate count NULL or non-1 labels as good/non-approve, silently understating rates
- 位置：`marvis/packs/data_ops/tools.py:527`（子系统：packs-misc-feature；验证结论：CONFIRMED）
- 细节：bad_rate compiles to avg(CASE WHEN try_cast(col AS DOUBLE) = 1 THEN 1.0 ELSE 0.0 END): a NULL label (this platform explicitly supports unlabeled rows -- there is a whole NaN-label confirmation gate elsewhere) or a non-numeric/alternate coding falls into ELSE 0.0 and inflates the denominator as a 'good'. A dataset with 20% unlabeled rows reports a bad_rate ~20% (relative) lower than the labeled-row bad rate every feature/modeling tool computes (they isfinite-mask labels), so the S6 analysis surface and the modeling surface disagree on the same slice. Same pattern for approval_rate (L532): NULL decisions count as non-approve. Correct SQL would exclude NULLs (avg over try_cast, or CASE ... WHEN NULL THEN NULL). tests/test_data_ops_slice_aggregate.py:74-93 verifies bad_rate only on a fixture with fully populated 0/1 labels, so it cannot catch this.
- 测试覆盖：none found

#### A2.5 [HIGH] Champion-selection evidence at the train_models gate cites the wrong metric and can claim the champion beat a runner-up it actually lost to
- 位置：`marvis/agent/renderers.py:383`（子系统：agent-render-service；验证结论：CONFIRMED）
- 细节：For binary targets the renderer hard-codes selector_key='oot_ks' / label '按 OOT KS' and builds the LT-11 champion-evidence line from it, but tool_train_models picks the champion via _pick_best_experiment on overfit-penalized TEST KS ('OOT is reported, not selected on — DOM-9', train_tools.py:443,501) or response_lift, and even emits the true 'selection_metric' field which the renderer ignores. Scenario: exp A wins on penalized test_ks with oot_ks=0.33, exp B has oot_ks=0.35 → the gate shows '最优 A（★；按 OOT KS）（依据：OOT KS=0.3300，较次优 B（0.3500）高 0.0200）' — wrong basis, and because the text uses abs(gap) with a fixed '高', it asserts A is HIGHER on OOT KS when it is lower. This is false evidence at the model-selection confirmation gate of a credit-risk workflow, and AUTO-mode LLM decisions read this same text.
- 测试覆盖：none found (tests/test_strategy_development.py:786 locks in the wrong '按 OOT KS' label with a fixture where the champion happens to also lead on oot_ks)

#### A2.6 [HIGH] is_start_validation_intent has no negation handling: '先别开始验证' / '不要开始验证' / '什么时候开始验证?' all launch the validation run

### A3. 专项核验：vintage 标签语义（上一轮对话遗留的头号疑点）
**结论：concern_confirmed（双怀疑者均 CONFIRMED，其中一位数值复现了 12 倍虚高）**

The concern is confirmed on both prongs: the double-count is reachable through a real product path, and the label-semantics contract is neither documented nor validated on that path.

Mechanics: compute_vintage_curve (marvis/validation/vintage.py) computes per-(cohort,MOB) bad counts in the first pass, then in the second pass (L110-124) sums those counts across MOBs and divides by a fixed cohort denominator (max sample_count/balance_sum across MOBs). This is a correct cumulative curve if and only if target_col means "newly became bad in this MOB" (each bad account flagged in exactly one MOB row). If target_col is an ever-bad snapshot ("bad as of this MOB" — the most common shape of loan performance long tables and of MOB observation flags), every bad account is re-counted at every subsequent MOB.

The repo itself proves the failure mode is real: the modeling pack's compute_vintage_report (marvis/packs/modeling/report_compute.py L113-162) melts mob_observe_cols — which it documents as cumulative snapshot flags — into exactly this long format, and its L155-159 comment says outright that using cum_bad_rate there "would re-accumulate and double-count the already-cumulative bads." That caller protects itself by reading metric='bad_rate'. But the strategy pack path (tool_vintage_curve → marvis/packs/strategy/vintage.py → metric='cum_bad_rate') applies the accumulation to a user-registered dataset with a bad_col picked by the LLM planner or user from task context. Nothing tells anyone what bad_col must mean per MOB: the manifest input schema has no description for bad_col, the workflow-template slot says only "Binary bad/default target column", and there is no validate step, no docstring, and no spec clause (the S3 performance-data contract covers id/snapshot/bucket columns for the future analysis pack, not vintage bad_col semantics).

The clip+warning is a masking mechanism on this path, in two independent ways. First, with realistic bad rates the inflated cumulative sum stays well below 1.0 (e.g. a 3% ever-bad cohort observed over 12 MOBs yields cum_bad_rate rising toward ~36% instead of ~3%), so the >1.0 guard never trips and the inflated curve — plus the vintage_summary 'deteriorating' trend computed from it — is presented as a decision_point gate payload with no signal anything is wrong. Second, even when the guard does trip, the strategy wrapper drops VintagePoint.data_quality_warnings on the floor: the VintageCurve contract has no warnings field, tool_vintage_curve's output schema has no warnings key, and the renderer shows only the clipped table. (vintage_summary_payload, which does aggregate warnings, is not used by any product caller — grep shows it only in the kernel and its own tests.)

Tests validate the incremental world only. tests/validation/test_vintage.py's "genuinely cumulative" tests use incremental fixtures; its one non-incremental test (L159-177) is deliberately extreme (2 loans, cum = 3/2) so it exercises the clip, and asserts clip+warning rather than a correct value — implicitly acknowledging the kernel cannot produce a correct answer from snapshot data. No test feeds sub-threshold ever-bad snapshot data, so the silent-inflation regime has zero coverage; tests/test_strategy_vintage.py compares wrapper output to kernel output (circular) and never touches semantics.

Net: the maintainer's own plan history shows cum_bad_rate was recently "fixed" from the old fake-cumulative bug into true accumulation, but the fix moved the correctness burden onto an input contract (incremental new-bad per MOB) that is unstated everywhere the user/LLM chooses the column, unenforced in code, silently violated by the one existing in-repo data shape (snapshot flags), and unprotected by the clip in the realistic regime. Minimal remediations would be: document/enforce semantics (e.g. a label_semantics={'incremental','snapshot'} parameter that switches accumulation off for snapshot data, as report_compute effectively does manually), a monotonicity-per-loan or per-MOB-bad-count heuristic red flag, and propagating data_quality_warnings through VintageCurve/tool output to the gate checklist.

- marvis/validation/vintage.py L110-124: second pass sums row['bad_numerator'] (per-MOB bad_count or bad_balance) across MOBs into cumulative_bad; correct only if each bad account appears as target=1 in exactly one MOB. No docstring or comment states this incremental contract anywhere in the file.
- marvis/packs/modeling/report_compute.py L155-160: comment 'mob_observe_cols are cumulative-by-construction snapshot flags: each column already means "bad as of this MOB"... ### A4. JOIN 子系统专项审计
（读文件 16 个）The JOIN path implements sample-table anchoring with left-join-only semantics, and its two headline invariants are genuinely ENFORCED, not advisory: the backend deletes the output and raises on any post-join row-count deviation (both fan-out and shrink), the engine re-checks and rolls back staged artifacts, and the confirmation gate is layered three deep (engine JoinNotConfirmedError, plan-level needs_confirmation on execute_join, and AUTO-mode risk flags that halt auto-confirm on all join gates) — I found no bypass via retry/cancel endpoints. Uniqueness, dedup and conflict detection are consistently computed in the transformed-key space that the actual JOIN uses, which is careful work. The weak spot is value normalization consistency at the edges: the executed JOIN, the diagnostics SQL, and the pandas fallback each normalize keys slightly differently, and I empirically confirmed two silent-corruption cases — empty/whitespace keys match in the real join while diagnostics exclude them (wrong feature values attached to missing-key rows with no error), and float64-stored IDs cast to scientific notation and match nothing while still passing the 1:1 assert. Confidence in the row-count and gating invariants is high; confidence in \"the matched values are the RIGHT values\" is materially lower, because nothing validates missing-key encodings, sentinels, or dtype provenance of key columns, and the boundary tests (blank keys, float keys, zero-padded CSV keys) don't exist.

#### [CRITICAL] Empty/whitespace keys match in the executed JOIN but are excluded from diagnostics — wrong feature row silently attached to every blank-key anchor row
- 位置：`marvis/data/backend.py:1135`
- 细节：_sql_transform/_sql_value_text (used by the real LEFT JOIN condition and by uniqueness/dedup) trim but do NOT nullif('') the key, while _sql_normalized_key (used by all match-rate diagnostics) wraps in nullif(..., ''), and the pandas fallback _transformed_key_value returns None for ''. Verified empirically: anchor rows with phone='  ' joined to a feature row with phone='' and received its score, while match_rate_for_method reported those rows as unmatched (1/3). With a unique feature key (or after first/last dedup collapses blank rows to one), the join stays exactly 1:1 so the post-join assert passes and the corruption is silent: every missing-key anchor row inherits one arbitrary row's features, and the user was shown a match rate that excluded them.
- 测试覆盖：none found

#### [HIGH] Float64-stored ID keys render as scientific notation in the JOIN's VARCHAR cast — 100% silent no-match; Python-fallback diagnostics disagree with the DuckDB join
- 位置：`marvis/data/backend.py:1136`
- 细节：Verified: an 18-digit idcard stored as float64 casts to '1.1010119900101123e+17' under trim(CAST(x AS VARCHAR)); the regex in _sql_value_text only strips '\.0+$', so the key never matches the string-side '110101199001011234' — all feature columns come back NULL while the 1:1 assert passes. Python _value_text renders the same value as the precision-lost integer '110101199001011232', so the Python match-rate fallback (feather features, unsupported hash algos) computes a different answer than the SQL join. Only CSV ingest has the long-id-as-string defense; parquet/feather/Excel uploads with float ID columns hit this. Partially mitigated by the shrink warning (match rate ~0 is at least visible), but a PARTIALLY-float column (mixed dtype coerced upstream) degrades match rate quietly.
- 测试覆盖：none found

#### [MEDIUM] CSV feature tables: diagnostics read all_varchar but the executed join reads read_csv_auto typed — zero-padded/typed keys can show high match then join to nothing
- 位置：`marvis/data/backend.py:517`
- 细节：match_rate_for_method(s) build feature keys via _duckdb_text_rel (all_varchar=true), preserving '001' as text; left_join builds the feature relation via _duckdb_rel (read_csv_auto typed), which infers INTEGER and casts back to '1'. A CSV feature keyed by zero-padded codes matches ~100% in diagnostics and 0% in the executed join (silent all-NULL features, 1:1 preserved). Latent in the product flow because the registry converts every upload to parquet, but left_join explicitly accepts CSV paths and the backend is a public seam used by tests/tools.
- 测试覆盖：none found (tests/test_data_backend.py exercises CSV joins but not zero-padded keys)

#### [MEDIUM] KEY_DICTIONARY substring matching over-matches column names — ETL timestamps like update_time qualify as 'date' join-key candidates
- 位置：`marvis/data/align.py:208`
- 细节：_matches_dictionary does normalized-substring containment, so 'update_time' (upDATE), 'validated_at', 'candidate_no' hit the 'date' family; 'hotel'/'telco' columns hit 'tel'. If both tables carry an export/update timestamp whose day-level values overlap >=50% (common when data is extracted in one batch), it passes MIN_KEY_MATCH_RATE and is AND-ed into the composite key — either quietly shrinking the true match (surfaced only as a low-match warning plus relaxation proposals the user must notice) or, when the timestamp is constant, adding a semantically meaningless key element the user is likely to rubber-stamp at the C2 gate.
- 测试覆盖：none found (tests/test_data_align* cover intended families, not substring false positives)

#### [MEDIUM] 'first'/'last' dedup silently switches from file order to whole-row lexicographic order when the feature has a column named file_row_number or a non-parquet path
- 位置：`marvis/data/backend.py:916`
- 细节：The UI note (join_gate_controller.js DEDUP_STRATEGY_NOTE) tells the user first/last keeps rows by file row order. _first_last_dedup_sql only uses parquet file_row_number when that name is free; otherwise it orders by ALL columns sorted ASC/DESC — deterministic but semantically 'row with lexicographically smallest/largest content', not first-in-file. **JOIN 路径的数据契约假设（未校验即信任的输入性质）：**
- Join keys are assumed precision-safe as strings: numeric IDs stored as float64 (parquet/feather/Excel uploads, or upstream-derived columns) silently break the join because DuckDB CAST(double AS VARCHAR) yields scientific notation ('1.1010119900101123e+17'); only the CSV ingest path (long_id_columns defense in csv_ingest) protects against this. Verified empirically.
- NULL is assumed to be the only missing-key encoding. Empty-string and whitespace-only key values are treated as real, matchable keys by the executed JOIN ('' = '' matches) but as missing by diagnostics (nullif) and by the pandas dedup fallback (None) — three layers disagree on what 'missing key' means.
- Sentinel key values ('N/A', '-', '0', '9999...') are assumed absent from key columns; nothing filters them, so identical sentinels on both sides attach an arbitrary feature row to every sentinel-key anchor row (join stays 1:1 and passes).
- The 'date' match method means exact calendar-day equality after canonicalizing through a fixed 6-format list (DATE_FORMATS); there is NO as-of / time-window / performance-snapshot join semantics anywhere in the join path (marvis/data/performance.py only validates YYYY-MM snapshot format). Any observation-window alignment is the user's responsibility before upload.
- Key-family detection relies on ASCII substring keywords ('date' in normalized name matches 'update_time', 'validate_flag'; 'tel' matches broadly); Chinese column names rely entirely on schema_infer's semantic_role. Wrong-family columns with >=0.5 empirical value overlap become proposed join keys.
- 'first'/'last' dedup assumes parquet physical file order is the business order; the UI note says 保留按当前文件行序, but when the feature table has a real column named 'file_row_number' (or a non-parquet path) the code silently switches to whole-row lexicographic content order.
- Registered datasets are assumed immutable on disk between propose and execute: spec.diagnostics (feature_key_unique, conflict counts) computed at propose time are trusted at execute time; the per-instance memo cache keys on (path, mtime_ns, size) only.
- hash:md5/sha256 matching assumes the hashed column was computed over exactly the trimmed, trailing-'.0'-stripped UTF-8 text with lowercase hex and no salt; differently normalized upstream hashes yield 0 match and are filtered empirically (MIN_KEY_MATCH_RATE), which is a hint mechanism, not a guarantee.
- confirm_join's engine-level 'confirmed' flag is set by an automatic plan step, not by the human: the actual human consent point is the needs_confirmation gate on execute_join. dedup_strategies can arrive pre-filled from the task_context slot (LLM-bound), in which case the gate shows diagnostics but no dedup picker; the strategy actually applied is only echoed after execution.

### A5. 建模方法学审计（对照"KS 封顶"判据）

**a) Is an OOT (time-based) split the default when a time column exists? Does providing time_col alone trigger it?** — 状态：`partial`

SEL-1 landed: when the sample carries a column whose lowercased FULL name exactly matches a loan-month alias (loan_month/apply_month/book_month/放款月/贷款月份/申请月份 — marvis/agent/modeling_setup.py:135-142), setup auto-sets split_config['oot_by_time'] and holds out the most recent ~20% of the ranked timeline (modeling_setup.py:267-270 joined path, :586-589 single-file path; carve in marvis/packs/modeling/prepare.py:166-184; covered by tests/test_modeling_api.py:487-507 and :843-883). BUT the task-level time_col field (marvis/domain.py:64) is NEVER passed into modeling setup — turn_handlers.py:659-673 calls build_modeling_proposal without it (task.time_col is only consumed by the vintage flow, turn_handlers.py:529). So the 2026-07-02 observation remains true in the general case: explicitly providing time_col does NOT trigger a time split unless the column happens to be alias-named (exact match, no substring — 'loan_month_2024' or 'obs_month' won't match), and a pre-existing split column is always passed through unchanged (modeling_setup.py:249-253). The only other trigger is an explicit split_config['oot_by_time'] via the make_split adjust path (marvis/agent/adjust_specs.py:117-119).

**b) Is ALL preprocessing fit on training folds only?** — 状态：`partial`

Largely fixed. Impute/normalize/cap fit their statistics on non-holdout rows only, default holdout ('test','oot'), and raise typed FitRequiresSplitError when no split column exists unless the caller explicitly passes allow_full_fit (marvis/packs/feature/tools.py:709-725; PREP-1). Numeric and categorical WOE encoding have the same gate (feature/tools.py:441-454). The scorecard recipe fits WOE maps on the train split only (marvis/packs/modeling/recipes/scorecard.py:47-60), the scorecard tune trial fits WOE on the trial's train (tune.py:959-989), the MLP impute+scale pipeline fits inside .fit on train/fold-train only (tune.py:1065-1070), and the accumulated chain is replayed at scoring time via the PREP-2 sidecar (prepare.py:79-83; train_tools.py:338-356). Remaining holes: (1) tool_onehot_encode fits its category mapping on the FULL frame including test/OOT (feature/tools.py:512-531) — structural, not label, leakage; (2) allow_full_fit is a single-flag opt-out with no audit trail beyond fit_split='full'; (3) the leakage screen and (via template wiring, see c/e) fine selection compute label statistics on train+test.

**c) Does fine/wrapper feature selection exist beyond coarse filters, and is it leak-free?** — 状态：`partial`

A multivariate refinement funnel exists between screen and tuning: IV floor + correlation dedup + optional VIF + top_k, in raw or WOE space with monotonic chimerge binning and a WOE sign check (marvis/packs/modeling/select.py:30-121, :151-265; template step 精选特征 marvis/orchestrator/templates/modeling.py:277-303). It is NOT wrapper selection: no RFE/stepwise/permutation/model-based iterative pruning exists anywhere in the repo (grep negative; marvis/feature/importance.py is a report-only gain ranking). Leak-freeness is engine-default-only: select_features excludes ('test','oot') by default with FitRequiresSplitError otherwise (select.py:124-148, tested at tests/test_modeling_select.py:205), but BOTH default modeling templates override it by passing the holdout_values slot — which is ['oot'] when an OOT exists (prepare_tools.py:133,189) — into select_features (modeling.py:293 and :581), so in the default plan IV/corr/VIF are fit on train+test. Also the default template effectively disables VIF (vif_max=1e9, modeling.py:299-303) and loosens corr_max to 0.95 vs engine default 0.8 (deliberate for tree recipes).

**d) Does hyperparameter tuning cover more than LightGBM? Are search spaces reasonable?** — 状态：`fixed`

tune.py now runs the two-stage (coarse 60% / fine 40%) seeded random search for ALL six binary recipes: lgb/xgb/catboost (40 trials each) and lr/scorecard/mlp (12 trials each) — DEFAULT_TRIAL_BUDGET tune.py:138-145, per-recipe samplers+runners tune.py:86-133, 643-1081, dispatch _recipe_search_hooks tune.py:1291-1440. Spaces are credit-risk-sensible: lgb lr 0.01-0.3 log, num_leaves 8-63, max_depth 2-6/-1, feature/bagging fraction, L1/L2 1e-8..10, min_child_samples 50-500; xgb eta/depth 2-8/subsample/colsample/min_child_weight/reg_alpha+lambda; catboost lr/depth 3-8/l2 1e-2..30; lr & scorecard C 0.01-10 (+scorecard max_bins 4-10); mlp alpha/lr_init/architectures. Tree trials early-stop on a 15% fold carved from train, not test (tune.py:1172-1182; recipes/common.py:87-126, SEL-4/TUNE-3); optional grouped k-fold CV scoring with a mean-minus-std robustness penalty (tune.py:179-317); per-recipe deterministic seed derivation (_common.py:150-156); train_models consumes per-recipe tuned params so the arena is fair (train_tools.py:290-352, TUNE-1/SEL-2); monotone constraints normalized identically to the train path (tune.py:1455-1489, TUNE-7).

**e) Is model selection/evaluation done on OOT/validation without test-set tuning contamination?** — 状态：`partial`

OOT itself is clean by explicit policy (DOM-9): tuning trial selection, champion selection (overfit-penalized test KS, train_tools.py:436-501), and the TUNE-4 champion refit (train+test only, OOT untouched — select_tools.py:217-284, train_tools.py:601-717) never consume OOT; early stopping watches a train-carved fold, not test (SEL-4). However the design deliberately reuses ONE test split for both per-recipe hyperparameter trial selection AND cross-recipe champion comparison, so the reported test KS is optimism-biased by construction (OOT is the only unbiased read, and it is report-only). On top of that designed reuse, two undesigned contaminations feed test information further upstream: the default templates fit select_features' IV/corr/VIF on train+test (modeling.py:293/:581 passing holdout=['oot']), and screen_features ranks candidates by KS computed on train+test (screen.py:196-207 dev mask excludes only OOT; deliberate for FS-4 diagnostics but it also drives the top_k=200 ranking). Post-refit headline metrics also include a test_ks measured on a 5% random carve of the refit training pool (train_tools.py:664-681), which is not comparable to the pre-refit test_ks.

**f) NaN-label gate: typed error + explicit drop-flag on ALL V2 pack paths (api and pipeline surfaces)?** — 状态：`partial`

The gate itself is thorough where it exists: all 9 training recipes call resolve_modeling_splits with config.drop_nan_labels (marvis/packs/modeling/recipes/{lgb,xgb,catboost,lr,scorecard,mlp,ensemble,lgb_regressor,lgb_multiclass}.py, e.g. lgb.py:39-40), tune_hyperparameters gates at tune.py:1152-1154, select_features at select.py:76-78, feature-pack WOE tools and strategy tools use require_labels_confirmed/resolve_labeled_frame (feature/tools.py:62, strategy/tools.py multiple), with per-split diagnostics and a legitimate fully-unlabeled-OOT carve-out (marvis/data/labels.py:81-152). The typed NanLabelNotConfirmedError carries to_detail() across the subprocess plugin boundary (marvis/plugins/subprocess_worker.py:640-650) and has a stable API error code (marvis/errors.py:46), and since every surface (plan driver / api / pipeline) executes the same pack tool functions through the same runner, both surfaces hit the gate. 17 tests in tests/test_label_nan_gate.py. Gaps: (1) tool_screen_features — the FIRST label-consuming step in the default plan — has NO gate: feature_ks silently isfinite-drops NaN-label rows (marvis/feature/metrics.py:19-25 via _finite_binary_pairs) with no count surfaced and no confirmation, so a NaN-heavy label passes the screen gate silently and only errors later at select/train; (2) no template/driver wiring exists for drop_nan_labels (grep of marvis/agent + marvis/orchestrator is empty), so confirmation relies on the generic gate-adjust path re-invoking the tool with the flag rather than an explicit purpose-built confirmation.

**方法学风险清单：**

#### [HIGH] Default templates leak test labels into fine feature selection (holdout=['oot'] overrides FS-2)
- 位置：`marvis/orchestrator/templates/modeling.py:293`
- 细节：Both modeling templates pass the holdout_values slot into select_features (line 293 and 581). That slot is ['oot'] whenever an OOT exists (prepare_tools.py:133/189, modeling_setup.py:312), which overrides select_features' safe default ('test','oot') via feature_tools.py:41's `if holdout else ("test","oot")`. Result: in every default run WITH an OOT, IV/correlation/VIF/top_k statistics are fit on train+test rows (select.py:144-148 excludes only 'oot'), so features are partially chosen for their test-split performance — the same test split that then selects the champion by test KS — while fit_split is misreported as 'train'. Perversely, runs WITHOUT an OOT get an empty list and fall back to the strict ('test','oot') default, so the leak only exists in the best-practice configuration.

#### [HIGH] Explicit task.time_col never triggers the time-based OOT split
- 位置：`marvis/agent/turn_handlers.py:659`
- 细节：build_modeling_proposal is called without task.time_col (only the vintage flow consumes it, turn_handlers.py:529). Auto time-OOT depends solely on an exact lowercase full-name match against the loan_month aliases (modeling_setup.py:135-142). A user who created the task with time_col='obs_month', 'loan_date', 'loan_month_2024' or any non-alias name gets a plain grouped 75/25 random split with NO OOT — the credit-risk-standard out-of-time evaluation silently degrades to n/a with only a chat note, and every downstream OOT metric/guardrail (max_oot_psi policy, OOT KS reporting) is vacuous. The maintainer's smoke-test observation ('time_col alone does not trigger a time split') is thus still true outside the alias-named happy path.

#### [MEDIUM] screen_features computes label statistics with no NaN-label gate
- 位置：`marvis/feature/screen.py:197`
- 细节：The leakage screen — the first label-consuming, user-confirmed step in the default plan — computes per-feature KS via feature_ks, which silently drops rows whose label is non-finite (_finite_binary_pairs, marvis/feature/metrics.py:19-25). With e.g. 40% NaN labels (immature vintages), screen rankings, the leakage gate (ks>=0.40) and top_k=200 selection are all quietly computed on the labeled 60% with no typed error, no drop-flag confirmation, and no nan count in the gate payload — violating the NaN-gate spec's intent for V2 packs; the user only hits NanLabelNotConfirmedError later at select/tune/train, after having confirmed a feature set whose statistics were computed on an undisclosed subset.

#### [MEDIUM] Post-refit headline metrics include a meaningless test_ks from a 5% scratch carve
- 位置：`marvis/packs/modeling/train_tools.py:672`
- 细节：The TUNE-4 champion refit relabels train+test as '__refit_train__' and carves a random 5% back out as '__refit_holdout__' purely to satisfy the non-empty-test contract; the code comment claims this slice is 'never reported', but compute_model_metrics computes test_ks/test_auc/psi on it and select_tools._apply_champion_refit returns ALL metric keys as the tool's headline `metrics` (select_tools.py:272-284, _is_metric_key filter). The final selected model therefore reports a test_ks measured on a tiny random slice of the data whose 95% siblings the model trained on — not comparable to the pre-refit test_ks shown beside it, and easy for the agent/report layer to quote as if it were the real held-out test KS.

#### [MEDIUM] Screen ranking KS computed on train+test lets test labels shape candidate ranking
- 位置：`marvis/feature/screen.py:196`
- 细节：The dev mask excludes only holdout_values=('oot',), so the KS that drives the top_k=200 ranking and the 0.40 leakage gate pools train+test labels (screen.py:196-207, 238). Deliberate for FS-4/FS-6 diagnostics (which need per-split KS anyway), but the pooled ranking means features are ordered partly by test-split discrimination before the champion is later selected on test KS — an additional, unmeasured optimism bias on the reported test metrics. Train-only ranking (with test used only for the ks_decay/split-shift diagnostics) would eliminate it at no information cost.

#### [LOW] tool_onehot_encode fits category mapping on the full frame including test/OOT
- 位置：`marvis/packs/feature/tools.py:519`
- 细节：Unlike impute/normalize/cap/WOE (all gated by _stat_fit_mask/_woe_fit_frame), onehot_encode has no split_col/allow_full_fit gate: the category set is derived from every row, so categories that only appear in test/OOT get columns. Structural rather than label leakage (no target involved), but it is the one transform exempted from the PREP-1 fit-on-train contract, and the resulting columns differ from what a train-fit encoder would produce at deployment time.

#### [LOW] Time-OOT carve is a row-rank quantile, not a time-span or size guarantee
- 位置：`marvis/packs/modeling/prepare.py:181`
- 细节：oot_by_time ranks the time column densely and takes ranked >= quantile(1-oot_size) over the per-row rank series. Months are atomic (all rows of a month share one rank, so no month straddles the cutoff — good), but with volume skew the actual OOT share can deviate wildly from oot_size: if the last month holds 50% of rows, quantile(0.8) lands inside/below that month's rank and OOT becomes ~50% of the sample, halving train; conversely a long sparse tail can make OOT tiny but many months wide. No warning is emitted about the realized share vs the requested one (the setup note prints the window and row count, but plan execution via make_split G1 relies on the user noticing).

#### [LOW] Default plan weakens the multivariate funnel it advertises (VIF off, corr loosened)
- 位置：`marvis/orchestrator/templates/modeling.py:299`
- 细节：The 精选特征 step ships with vif_max=1e9 (matches correlation.py's sentinel cap, so the VIF gate can never trip) and corr_max=0.95 vs the engine default 0.8. Documented as deliberate for tree recipes, but the same funnel output also feeds lr/scorecard candidates in the multi-algorithm arena, where multicollinearity genuinely hurts — a scorecard trained on a 0.95-correlated feature set gets unstable/sign-flipped coefficients (only the warn-level sign check catches it, and only in WOE space, which the default template does not use: space='raw').

**建模路径的数据契约假设：**
- Time columns must be lexicographically sortable if textual: oot_by_time ranks raw values with dense rank (prepare.py:176-183), so '2025-9' > '2025-10' style month strings would silently produce a wrong chronological OOT; the code comment assumes zero-padded 'YYYY-MM' strings.
- Auto time-OOT assumes the loan/apply month column is named exactly one of: loan_month, apply_month, book_month, 放款月, 贷款月份, 申请月份 (lowercase full-name match, modeling_setup.py:135-142); any other name means no OOT by default.
- holdout_values has two different meanings that share one template slot: 'splits excluded from the leakage screen' (screen, default ('oot',)) vs 'evaluation splits selection must never fit on' (select_features, default ('test','oot')) — the templates wire the same ['oot'] value into both.
- An entirely unlabeled OOT is treated as a legitimate scoring-only split (labels.py:128-134), while a partially labeled OOT triggers the NaN gate; label metrics assume labels are exactly {0,1} after numeric coercion (non-numeric labels are a hard error, NaN handled by the gate; values like 2 would be silently neither good nor bad in KS counts).
- Sample-weight columns are assumed strictly positive and fully non-null at train time (recipes/common.py:381-394 raises otherwise), but tune.py's _sample_weight allows zeros (only negatives rejected, tune.py:616-628) — tune and train enforce slightly different weight contracts.
- prepare_modeling_frame assumes the split frame and source frame share row order for the make_split sample-analysis month/channel table (prepare_tools.py:198-214, positional zip).
- Group-based anti-leakage carving assumes identity columns are detectable by name tokens (cust_id/user_id/身份证/... modeling_setup.py:551) and silently degrades to per-row splitting when the group column was dropped before modeling (prepare.py:51-58).
- Reproducibility contract is asymmetric by design: lgb tuning trials are only bit-identical on machines with the same core count (DEFAULT_TUNE_NUM_THREADS=0, defaults.py:22), while xgb/catboost/lr/scorecard/mlp trials and all direct train paths are cross-machine deterministic (num_threads=1).

## Part B：所有者五问

### Q1. 你现在最大的盲点是什么？你忽略了什么？

**最大的盲点：你建立了一套极其严格的质量体系，但它的每一层都在同一个方向上看——向内（代码 vs 设计），没有一层向外（设计 vs 数据）。** 2008 个测试、三轮全面审查、179 项 backlog、双收官循环、typed gate、确定性内核——这套过程的诚实度我们这轮实证过了（15/15 落地核验通过）。但 A2 那 8 条确认级 bug 全部从这套体系里穿过去了，原因不是体系执行得不严，而是体系的结构性视角：测试用的合成数据是按代码的假设造的，审查对照的是 spec，backlog 核验对照的是 claim。当假设本身错了，所有层一起失明。你在"有没有 bug（代码是否按设计工作）"上投入了可能 90% 的质量精力，在"设计对不对（给定真实数据的长相）"上投入接近零——而后者才是风控平台的命门。

具体被忽略的四样东西，按代价排序：

1. **语义契约无人拥有。** 每个 tool 输入列"是什么意思"——`bad_col` 是快照还是增量、NULL 标签算什么、-999 是不是哨兵值、面板数据的粒度是贷款还是贷款×月——这些语义没有任何一层负责声明、校验或传递。你为 NaN 标签修了一个非常好的强制确认门（DOM-1，落地核验通过），但那是对这个"类"打的一个"点"补丁。A2 的 #1/#2/#3/#4/#7/#8 六条全是同一个类的不同实例。你修 bug 的速度不可能快过这个类产生实例的速度，除非把类本身消灭（见 Q3）。

2. **证据层被当成了展示层。** 你的整个安全模型是"人在环上"：确定性内核算数 → gate 呈现证据 → 人确认。这个链条的强度等于最弱一环，而你一直把渲染层归类为"前端/口味"问题（你的记忆偏好里甚至专门写过"勿让审查盖过用户口味"——那是对的，但它只该管美学）。A2.5 证明渲染层能对着正确的计算结果向你呈现错误的证据、甚至断言赢了实际输掉的对比——这时人的确认就是盖橡皮章，AUTO 模式下 LLM 读的也是这段假证据。**gate 上的每一个字都是安全部件，不是 UI。**

3. **过程指标替代了结果指标。** 你追踪 tests passed、backlog ✅ 数、审查轮数——全是过程指标。没有一个持续运行的数字回答"agent 在一份真实数据上端到端产出的模型/报告是否正确"。你的核心判据"同样本同标签 KS 封顶"存在于文档里，但仓库里没有一个 harness 定期用真实（或公开）数据集跑这个判据。六旅程 smoke 是合成数据的快乐路径。换句话说：**你有 2000 个测试，但回归回路里有 0 份真实数据。**

4. **你自己就是最后一个测试盲区。** 单机单用户，你既是开发者又是唯一用户：你喂的数据恰好符合你写代码时的假设，你说的中文恰好是意图正则的训练集（A2.6 的"先别开始验证"→开跑，对你不会发生，因为你不会这么说；换任何一个人就会）。平台从未接受过一个"不知道代码假设什么"的人的输入——而这正是 Q5 里"赋能业务的人"场景的日常。

### Q2. 如果这个项目 3 个月后失败了，最可能的原因是什么？

按概率排序，前两条是真正的威胁：

**第一名：信任破产（数字错一次，全平台数字失去无罪推定）。** 机制：某次真实数据分析中，一个 headline 数字被外部事实戳穿——vintage 曲线比财务口径高一个数量级、EL 比拨备大 12 倍、bad_rate 和另一张报表对不上 20%。风控工具卖的不是功能，是"这个数字可以直接进决策/汇报"的信任。信任的破产方式不是崩溃而是安静的降级：从"直接用"变成"用之前先在 Excel 里复核一遍"——一旦到了这一步，平台就从生产力工具变成了额外负担，弃用只是时间问题。我把它排第一，因为 A2 的 8 条确认 bug 里 6 条正好是这个形状，而且全部偏向"真实数据比测试数据脏"的方向——也就是说，**按现状接入第一份真实业务数据时，踩中至少一条的概率不是"可能"而是"接近必然"**（sentinel -999 在征信数据里是常态；快照型 vintage 面板是最常见的表现数据形状；无标签行在未成熟账龄里必然存在）。

**第二名：收官循环结构性不收敛 → 单人项目的动能耗尽。** DoD 第 11 条要求"循环直到一轮 review 无新 critical/high"。但这一轮 120 个 agent 在一棵"全绿、双 review 已过"的树上仍然产出了 8 条确认级——发现能力是随投入弹性增长的，这个出口判据在结构上永远达不到。单人项目最常见的死法不是技术失败，而是在还债循环里耗尽热情：3 个月后 backlog 从 179 变成 240，V2.1 还没让任何真实业务受益，动能归零。**这两条会共振**：为了防第一名你会更用力地审查（喂养第二名），而审查永远审不出数据契约问题（防不了第一名）——这是个陷阱结构。

破局判据其实很简单，两条都指向同一个动作：**把出口判据从"审查无新发现"换成"N 份真实/公开数据集端到端跑通且数字经外部对账正确"**。审查轮数封顶，真实数据 smoke 作为收官出口。修 A2 的 8 条 + 给 6 个最危险的语义假设加校验门，大约是两周量级的工作，比再来一轮全面审查便宜得多、有效得多。

第三名及以后（3 个月内概率较低但值得记录）：复杂度墙（175k LOC 单人维护，复发 bug 是早期信号——但 15/15 落地核验说明目前还 hold 得住，这是 6-12 个月尺度的威胁）；"不失败但也不成功"（平台永远停在开发台，产出物没有接进任何真实决策流——见 Q5 的部署缺口）。

### Q3. 加一个你没要求、但能让项目行业领先的功能？

**"可信数字层"（Trusted Numbers）：平台呈现给人的每一个决策数字，都自带独立复算 + 血缘 + 一键重放。** 三个组成部分：

1. **双路对账（复式记账）。** 每个进入 gate/报告的 headline 指标（KS、match_rate、vintage、EL、bad_rate……）由两条独立实现各算一遍——平台里本来就同时存在 DuckDB/SQL 路径和 pandas 路径，A4 证明了它们目前是"分裂风险"（同一个 join 两条路算出不同答案、静默）；这个功能把劣势直接翻转成资产：**强制对账，不一致 = 阻断性红旗**。空白键 bug、科学计数法 bug、NULL-as-good bug 在双路对账下全部当场现形，因为两条路对脏数据的处理分歧正是 bug 所在。
2. **数字血缘。** 每个呈现的数字携带最小溯源元组：数据集指纹、代码版本、参数、种子、上游 join/预处理链摘要。gate 上的数字可展开看"这个 36% 是从哪张表的哪列、经过什么变换算出来的"——A2.1 那种"36% 其实是 3%"的错误，用户顺着血缘看到"逐 MOB 累加了一个快照列"就能自己抓住。
3. **一键重放。** 任何历史报告里的任何数字，一键在当前代码上重算并 diff。这是你已有的确定性内核（INV-1）、证据信封、artifact staging 的自然完成态——你已经为它付了 80% 的地基成本，只差把最后一层楼盖起来。

为什么是它而不是别的：(a) 它精确瞄准本轮发现的唯一幸存 bug 物种，是"消灭类而不是修实例"的那个动作；(b) 它把行业里 LLM agent 的根本疑虑——"AI 给的数字我敢用吗"——从营销问题变成机制问题，**没有任何竞品（内部平台或商业产品）做到"每个数字自带独立复核"**，这在风控这个"数字即产品"的领域是护城河级差异；(c) 对监管叙事（模型风险管理、审计可追溯）是降维打击。落地可以渐进：先挑 gate 上的 5 个最高危数字做双路对账（join match 统计、vintage、EL、KS、bad_rate），一个月内可见效。

次选（一句话）：**公开数据集基准竞技场**——把 GiveMeSomeCredit / Home Credit 等公开信贷数据集接成 nightly harness，agent 全流程跑 vs 专家基线 KS 记分牌，把你的"KS 封顶"判据从口头判据变成持续回归——它同时是 Q2 破局判据的具体实现。

#### Q4.1 重复实现地图（最重要——它们是 INV-1"确定性一致指标"的直接敌人）

### Q5. 现在的 agent 能覆盖信贷风控全流程开发吗？缺什么？能赋能一个懂业务的人独立负责一条业务线吗？

#### Q5.2 "赋能一个懂业务的人独立负责一条业务线"——分两半回答

**开发职能：接近可以。** 一个懂业务、懂方法论概念但不写代码的人，今天能在平台上走完 数据接入→join→特征→建模→验证→策略→PMML 交接 的全过程，且产出是真的（15+10 项 backlog 双轮核验、方法学审计支撑：预处理 train-only、调参覆盖六族、OOT 政策隔离、每步强制 gate）。这大约等于一个初级建模团队的产出能力。**但有两个前提**：① 数据必须以"平台假设的形状"进来——而 Q1 论证过，语义契约问题对"懂业务不懂数据工程"的人恰恰最致命，他们是最不可能发现 vintage 虚高 12 倍的人群。**可信数字层（Q3）是"一人业务线"的前置条件，不是锦上添花**。② 定坏口径、观察期/表现期设计——整个风控建模最重要的决策——目前必须在平台外完成（阶段 2 blocker）。

**运营职能：不行，且不是差一点。** 部署（无评分 API、无影子运行）、监控闭环（无调度、无推送）、反欺诈（全缺）、审批记录（单人自批）——这四块决定了模型和策略造出来之后，`没有外部系统和人手就无法安全地跑在真实业务上`。今天平台的产出物边界是 PMML/决策表/报告，不是运行中的决策服务。

#### Q5.3 通往"一人业务线"的最短路径（按 ROI 排序）

1. **标签构造与成熟度工具**（补阶段 2 blocker）：从还款/DPD 长表构造 0/1 目标（观察期+表现期+定坏口径参数化），附 vintage 成熟度检查（"这个 cohort 的表现期闭合了吗"）和 roll_rate→定坏口径的桥接（两个组件都已存在，只差连线）。这是全清单里唯一既是 blocker 又完全在平台现有能力射程内的项。
2. **批量评分 API + 影子运行**（补阶段 7）：先不做实时——一个 `/score` 批量端点 + "新旧策略对同一批新数据双跑出对比报告"的影子模式，就能支撑月度/周度决策刷新的业务线形态。
3. **监控调度 + 推送告警**（补阶段 8）：cron 触发 monitor_run + webhook/邮件出口。没有它，"早期预警"依赖人记得点按钮。
4. **征信报文解析器**（补阶段 3）：人行二代报文→特征表。工作量大，但它决定平台能不能吃到最重要的外部数据源。
5. **反欺诈建议显式宣布不做**（产品选择，写进 DoD 第 7 条的产品选择清单）：名单/设备/图谱是另一门学科，与其半做不如明确边界。
6. maker-checker 治理：单机单用户是既定架构决策，不必推翻；但可以低成本补"审批留痕导出"（谁在何时确认了什么 gate、依据什么证据——审计路由已有 80% 地基）。

### modeling-delivery（读 12 个文件）

This slice is the post-training delivery and governance surface of the modeling pack: PMML export with a pypmml round-trip validator (artifact.py), V1 validation handoff and challenger/backtest packages with a generated self-contained scoring notebook (handoff.py), the approval package / model card / monitoring policy / champion-vs-challenger comparison writer (delivery_tools.py), apply-model-to-new-data scoring plus drift monitoring against a training-time baseline snapshot (monitor_tools.py, scoring.py), and the shared dataclass contracts. Engineering quality is high — transactional artifact writes with rollback, explicit capability gating for non-PMML algorithms, careful PREP-2 preprocessing-replay opt-in, and matching WOE bin-boundary semantics between native scoring and the PMML expression (verified against assign_bins). The weakest area by far is monitoring self-consistency: the baseline snapshot is taken on raw, uncalibrated scores over the post-preprocessing modeling frame, while monitor_run compares calibrated scores and raw-space feature values against it, and uses train_ks (not OOT) as the degradation reference — three independent ways the flagship 'green/amber/red' verdict fires falsely on perfectly healthy models, none exercised by tests. The champion-comparison direction inference also inverts weighted PSI, feeding an over-optimistic verdict into the human approval package. The delivery/handoff plumbing itself (packages, notebooks, tasks, audits) looks solid and is comparatively well tested.

**跳过/部分读取：** marvis/packs/modeling/train_tools.py (read only lines 700-920, the baseline-snapshot code this slice depends on; file belongs to the training slice)；marvis/packs/modeling/calibrate_tools.py (grep-verified only that it never recomputes baseline_distributions; belongs to another slice)；marvis/feature/binning.py (read equal_frequency/assign_bins sections only)；marvis/notebook_contract.py (read head/tail-cell injection section only)

**中低风险发现（未独立验证）：**

- **[medium] ks_drop/auc_drop use train_ks/train_auc as the development reference, so any normally-overfit model false-alarms on live data**（`marvis/packs/modeling/monitor_tools.py:415`）
  _monitor_run_label_checks reads dev_ks = metrics.train_ks and dev_auc = train_auc — the in-sample fit split — while the threshold doc (monitor_tools.py:119-121) says the check flags drops below 'the model's own development-period reading'. Training gates elsewhere accept overfit_train_test_gap up to 0.08 warn / 0.12 fail. Scenario: LGB with train_ks 0.52, test_ks 0.45, oot_ks 0.44 (passes selection); a labeled live sample performing exactly at OOT level (KS 0.44) yields ks_drop = 0.08 → amber, and at KS 0.41 → 0.11 → red 'performance degradation', even though live discrimination equals development OOT. The natural reference is oot_ks/test_ks; using train systematically overstates degradation for every tree model.
  测试覆盖：none found for the failure mode (tests/test_modeling_monitor.py:57 green test uses a linearly-separable LR with ~zero train/test gap, so the inflated reference never trips)

- **[medium] weighted_psi_* direction inverted in champion/challenger comparison — a challenger with worse weighted PSI is reported 'improved'**（`marvis/packs/modeling/delivery_tools.py:720`）
  _metric_better_direction only checks lowered.startswith('psi_'), but weighted models carry metrics named weighted_psi_test_vs_train / weighted_psi_oot_vs_train (contracts.py:52-53), which _is_metric_key admits via the 'weighted_' prefix. They match none of the lower-is-better substrings and default to 'higher'. Scenario: champion weighted_psi_oot_vs_train=0.05, challenger=0.30 (materially less stable under the selection-driving weighted 口径) — delta +0.25 gets verdict 'improved', inflates improved_count, and can leave comparison status 'pass' with recommendation 'Challenger 不弱于 Champion', which flows verbatim into the approval package and model card a reviewer signs off on. (*_ci_std metrics are mis-directed the same way.)
  测试覆盖：none found (tests/test_modeling_pack.py challenger tests assert champion identity/artifact paths only, never per-metric verdicts, and never include weighted metrics)

- **[low] post_training_action hands a stale artifact to handoff, re-exporting PMML from the sample dataset and hard-failing (not skipping) when the training dataset is gone**（`marvis/packs/modeling/delivery_tools.py:192`）
  tool_post_training_action fetches the artifact once (line 158). export_pmml updates pmml_path only in the DB via _pmml_path (lines 1794-1796); the in-memory artifact still has pmml_path=None, so handoff_to_validation/create_challenger_backtest re-run _ensure_pmml_path (handoff.py:312-326) and regenerate+overwrite {artifact.id}.pmml from the SAMPLE dataset instead of reusing the just-exported one — duplicated heavy export, and for scorecards the delivered PMML's schema-sample comes from whichever export ran last. Worse, _pmml_path resolves the original training dataset via runtime.registry.get(experiment.config.dataset_id) (line 1784); if that derived dataset was pruned after training, the whole delivery tool raises out of registry.get instead of degrading the export action to 'skipped' as the tool's docstring promises — no approval package/model card gets written at all.
  测试覆盖：none found for the success-path combination (tests/test_modeling_handoff.py and tests/test_modeling_pack.py:2347 only exercise combined actions on unsupported algorithms where everything is skipped)

- **[low] mark_validated_from_validation_task marks an experiment 'validated' from ANY succeeded task, with no linkage check to the artifact or handoff**（`marvis/packs/modeling/handoff.py:140`）
  The function looks up validation_task_id, checks only task.status in {SUCCEEDED, REVIEW_REQUIRED}, and flips artifact.experiment_id to 'validated' — it never verifies the task is a validation task created for this artifact (model_version/source_dir are ignored). Scenario: a caller passes the id of an unrelated succeeded task (or the challenger-backtest task of a different artifact) and the experiment is promoted to 'validated', which _previous_selected_champion_reference (delivery_tools.py:617) then treats as champion-eligible. Currently unreachable from production code (no callers outside tests), so this is latent API risk rather than a live defect — but it is exported in __all__ as a public entry point.
  测试覆盖：tests/test_modeling_handoff.py:787 covers the happy path only; no test passes a mismatched task id

**数据契约假设：**
- Handoff/backtest sample dataset (handoff.py:47,164) is assumed to contain every artifact.feature_list column (or the raw inputs of the preprocessing chain) plus target/split/time columns with training-time semantics; the package build copies it verbatim and never validates schema, so a wrong sample only surfaces later when the V1 harness executes the notebook. score_col default 'pred' and time_col default 'apply_month' (handoff.py:381-386) are pure conventions read from params.
- monitor_run's scored_dataset_id branch (monitor_tools.py:209-228) trusts that score_col contains THIS artifact's PD in higher-is-riskier orientation; passing scorecard points, another model's scores, or a raw-score column on a calibrated artifact silently produces meaningless PSI/KS-drop with no provenance cross-check.
- Labels are assumed to be literally 0/1 (monitor_tools.py:398 np.isin(raw_labels,[0.0,1.0])); portfolios coded 1/2, Y/N, or probabilities silently degrade every KS/AUC check to 'n/a' instead of erroring.
- Baseline snapshot assumes the training dataset IS the post-preprocessing modeling frame and that split_col contains config.split_values['train'] (default 'train') (train_tools.py:754); monitor_run then assumes new data is RAW pre-preprocessing data — the two conventions contradict for feature CSI (see risk).
- Auto champion resolution (delivery_tools.py:611-648) assumes any earlier 'selected/validated/delivered/approved/champion' experiment in the same task is metric-comparable — it never checks dataset_id, target_col, or target_type, so a prior experiment on a different target in the same task silently becomes the 'champion' baseline; ordering relies on lexicographic ISO created_at strings.
- _metric_better_direction (delivery_tools.py:717) infers better-direction purely from metric-name substrings (psi_/rmse/mae/logloss/gap...); any metric not matching those conventions (weighted_psi_*, *_ci_std, custom names) is assumed higher-is-better.
- The generated scoring notebook (handoff.py:582-644) assumes the V1 harness injects RMC_SAMPLE_PATH via a head cell (marvis/notebook_contract.py:318) and executes with cwd = the material dir (model/calibration files are bare relative names); neither is enforced in the notebook itself.
- Monitoring-policy threshold overrides are keyed by metric name and matched against experiment.metrics keys (delivery_tools.py:415-421); a typo'd or non-existent metric silently becomes status 'missing' (overall warn) rather than a configuration error.
- sample-weight governance (delivery_tools.py:519-526) detects a weight column via three alias param keys and only records its presence; weight positivity, scale, and business meaning are never validated anywhere in the delivery path.

### modeling-tune-select（读 7 个文件）

This slice is the hyperparameter search (two-stage coarse/fine random search generalized to lgb/xgb/catboost/lr/scorecard/mlp, with optional grouped k-fold CV scoring), statistical feature selection (IV/corr/VIF in raw or WOE space with anti-peek holdout masking), champion comparison/selection with policy gates and post-selection train+test refit, post-hoc probability calibration and segment-value diagnostics, plus the experiment store and scenario templates. The code is unusually well-documented (each fix carries a TUNE-n/SEL-n/DOM-n rationale) and the core search loop, trial scoring, policy machinery, and NaN-label gating are solid and heavily tested. ### data-layer（读 18 个文件）

This slice is the deterministic data foundation: it normalizes uploads to parquet (encoding fallbacks, long-id and Excel header defenses), profiles columns from samples into fingerprints and semantic roles, empirically aligns join keys across raw/hashed/date representations, and executes anchor-left-joins behind a strict 1:1 row-count assertion with two-level dedup and typed confirmation gates (NaN labels, dedup-required, fan-out, performance-frame contract). In short: the guards are solid, but what the guards are told to guard is sample-based, name-based, and unvalidated - that is where I am least confident.

### routers（读 21 个文件）

This slice is the HTTP adapter layer: thin FastAPI routers that translate requests into repository/engine calls for dataset ingestion and the join engine, plan orchestration, the LLM validation agent, report generation, plugin/draft management, and audit/memory browsing. Overall it is well built — consistent bounded pagination, a single-active-job invariant enforced via a DB unique index, transactional artifact staging with rollback, streamed uploads with an authoritative byte-count guard, and optimistic-concurrency (If-Match/revision) on report fields. The real weaknesses are at trust boundaries and in cross-router guard consistency: the join-confirm endpoint builds KeyPair domain objects directly from raw client dicts (including a client-dictated match_rate that the engine then treats as measured), join plans can be re-confirmed/mutated after execution, cancel endpoints differ in whether they check for an active job (allowing a stale cancel to poison the next run), delete_task's path-escape guard catches the wrong exception type, and the generic artifacts route serves the same report file that reports.py deliberately gates behind conclusion confirmation.

### memory-plugins（读 22 个文件）

agent_memory/ is a SQLite-backed cross-task memory store: typed MemoryCandidates gated by a regex/allowlist policy, redacted, deduped and audited; a DistillationEngine groups active entries by scope_key and merges them into MemoryDistillations (optionally LLM-summarized) with net support = distinct positive tasks minus distinct negative-feedback tasks; an EvolutionManager upserts/supersedes/rolls back distillation heads; ConsolidationScheduler fires distillation on hook events (async daemon threads, 300s throttle marked only after completion); retrieval scores raw entries plus distillations into bounded prompt packets. plugins/ is the V2 tool runtime: manifest parsing/validation, DB-backed registry, zip installer with checksum, hook dispatcher, and a subprocess ToolRunner that JSON-schema-validates inputs/outputs, derives seeds for stochastic tools, enforces an RSS monitor plus CPU/file rlimits, and monkey-patches file/network/process access inside the worker. Overall the code is careful and unusually well-audited (transactional distillation replace with rowcount guards, protocol version handshake, redaction at every persistence point), with strong single-threaded test coverage.
